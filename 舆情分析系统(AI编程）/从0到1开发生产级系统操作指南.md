# 从0到1开发生产级系统 — Claude全家桶操作指南

> 适合人群：完全没用过 Claude Code 的小白
> 目标：照着这份指南，能独立开发并上线一个生产级系统

---

## 一、先搞清楚你手里有什么

这套"全家桶"本质上是一套**已经配好的 AI 编程工作流**，核心组件如下：

| 组件 | 在哪里 | 是什么 |
|------|--------|--------|
| **Agents（智能体）** | `~/.claude/agents/` | 专职 AI 助手，自动在合适时机启动 |
| **Skills（技能）** | `~/.claude/skills/` | 特定任务的操作手册，用 `/技能名` 调用 |
| **Commands（命令）** | `~/.claude/commands/` | 斜杠命令，快速触发工作流 |
| **Rules（规则）** | `~/.claude/rules/` | Claude 必须遵守的编码规范，自动生效 |
| **Hooks（钩子）** | `settings.json` | 自动化触发器，比如保存文件后自动格式化 |

**你不需要手动管理这些**，它们已经配好了，Claude 会自动调用。

## 二、开发一个系统的完整流程

### 阶段0：准备工作（开始前必做）

**1. 确认 Claude Code 已安装**
```bash
claude --version
```

**2. 新建项目目录，进入后启动 Claude**
```bash
mkdir my-project
cd my-project
claude
```

**3. 初始化 Git（必须，很多功能依赖它）**
```bash
git init
```

---

### 阶段1：立项规划

> 目标：把你的想法变成一份清晰的实现计划

**你要做的：** 用自然语言描述你想做什么，然后运行：

```
/plan 我想做一个[你的产品描述]，需要[核心功能列表]
```

**Claude 会自动：**
- 调用 `planner` agent（用 Opus 模型，最强推理）
- 分析需求，拆解成多个阶段
- 列出每个步骤涉及的文件、风险、依赖
- **等你确认后才开始写代码**

**示例：**
```
/plan 我想做一个用户管理系统，需要注册登录、角色权限、用户列表
```

**收到计划后：**
- 仔细看每个阶段，有问题就说 `modify: 把第2阶段改成...`
- 没问题就回复 `yes` 或 `proceed`

---

### 阶段2：技术选型与架构

> 目标：确定用什么技术栈，搭好项目骨架

**告诉 Claude 你的技术偏好：**
```
我用 Next.js + TypeScript + Supabase，帮我搭建项目骨架
```

**如果不知道选什么技术，直接问：**
```
我要做一个[产品类型]，推荐什么技术栈？帮我分析优缺点
```

**Claude 会自动：**
- 调用 `architect` agent 做架构决策
- 创建项目结构
- 生成 `package.json`、配置文件等

---

### 阶段3：功能开发（核心循环）

> 这是你花时间最多的阶段，遵循 **计划 → 测试 → 实现 → 审查** 的循环

#### 每个功能的标准开发流程：

**Step 1：规划这个功能**
```
/plan 实现用户注册功能，包括邮箱验证和密码加密
```

**Step 2：用 TDD 方式开发（先写测试，再写代码）**
```
/tdd 实现用户注册的 API 接口
```

Claude 会：
1. 先写测试（RED — 测试会失败）
2. 写最少的代码让测试通过（GREEN）
3. 优化代码（REFACTOR）
4. 确保覆盖率 ≥ 80%

**Step 3：代码审查**
```
/code-review
```

Claude 会检查：
- 有没有安全漏洞（SQL注入、XSS等）
- 代码质量（函数太长、嵌套太深等）
- 有没有 console.log 没删

**Step 4：E2E 测试关键流程**
```
/e2e 测试用户注册到登录的完整流程
```

#### 重复以上4步，直到所有功能完成。

---

### 阶段4：安全加固

> 上线前必做，不能跳过

```
/code-review
```

重点检查：
- 所有密钥是否在 `.env` 文件里（不能硬编码在代码里）
- 用户输入是否都做了验证
- API 接口是否有认证保护
- 数据库查询是否用了参数化查询

**如果发现安全问题，Claude 会自动调用 `security-reviewer` agent 处理。**

---

### 阶段5：上线部署

**构建检查：**
```
/build-fix
```
如果有报错，Claude 会自动调用 `build-error-resolver` agent 修复。

**最终代码审查：**
```
/code-review
```

**提交代码：**
```
/commit
```
Claude 会自动生成符合规范的 commit message。

## 三、常用斜杠命令速查表

| 命令 | 什么时候用 | 做什么 |
|------|-----------|--------|
| `/plan` | 开始任何新功能前 | 生成实现计划，等你确认再动手 |
| `/tdd` | 写每个功能时 | 先写测试再写代码，保证质量 |
| `/code-review` | 写完代码后 | 检查安全和质量问题 |
| `/e2e` | 功能完成后 | 测试完整用户流程 |
| `/build-fix` | 构建报错时 | 自动修复编译错误 |
| `/refactor-clean` | 代码乱了 | 清理死代码、重构 |
| `/save-session` | 要休息时 | 保存当前进度，下次继续 |
| `/resume-session` | 重新开始时 | 加载上次进度 |

## 四、Agents 自动启动规则

你不需要手动调用这些 agent，Claude 会在合适时机自动启动：

| Agent | 自动启动时机 |
|-------|------------|
| `planner` | 你描述复杂功能时 |
| `tdd-guide` | 你要写新功能或修 bug 时 |
| `code-reviewer` | 你写完/改完代码后 |
| `security-reviewer` | 涉及登录、支付、用户数据时 |
| `build-error-resolver` | 构建失败时 |
| `architect` | 需要架构决策时 |

## 五、新手必知的3个坑

### 坑1：Context Window 耗尽

**症状：** Claude 开始忘记之前说过的内容，回答变得奇怪。

**解决：**
```
/save-session    # 先保存进度
/compact         # 压缩上下文
```
或者直接开新对话，用 `/resume-session` 加载进度。

**预防：** 每完成一个功能阶段就 `/save-session` 一次。

---

### 坑2：密钥泄露

**永远不要这样写：**
```javascript
const apiKey = "sk-abc123..."  // 危险！会被提交到 git
```

**正确做法：**
```javascript
const apiKey = process.env.API_KEY  // 从环境变量读取
```

在项目根目录创建 `.env` 文件存放密钥，并确保 `.gitignore` 里有 `.env`。

---

### 坑3：跳过测试直接写代码

很多人觉得 TDD 麻烦，想直接写代码。但这会导致：
- 后期改一个地方，不知道哪里会坏
- 上线后出 bug 很难定位

**坚持用 `/tdd`**，前期多花20%时间，后期省80%调试时间。

## 七、一个完整项目的时间线示例

以"用户管理系统"为例：

```
Day 1：立项
  /plan 用户管理系统需求分析
  → 确认计划
  → 搭建项目骨架

Day 2-3：核心功能
  /tdd 用户注册 API
  /tdd 用户登录 API
  /tdd 权限中间件
  /code-review

Day 4：前端页面
  /plan 前端页面规划
  /tdd 注册页面组件
  /tdd 登录页面组件
  /e2e 注册登录完整流程

Day 5：收尾上线
  /code-review  （全面安全检查）
  /build-fix    （修复构建问题）
  /commit       （提交代码）
  部署到生产环境
```

---

## 八、遇到问题怎么办

**Claude 给出的代码有 bug：**
直接描述问题，Claude 会自动调用相关 agent 修复。

**不知道下一步做什么：**
```
我现在完成了[已完成的内容]，下一步应该做什么？
```

**代码越来越乱：**
```
/refactor-clean
```

**构建一直报错修不好：**
```
/build-fix
```
如果还不行，把完整错误信息粘贴给 Claude。

---

## 九、核心原则（记住这3条）

1. **先计划，再动手** — 每个功能都用 `/plan` 开始，不要直接让 Claude 写代码
2. **测试先行** — 用 `/tdd`，不要跳过测试
3. **定期保存** — 每完成一个阶段用 `/save-session`，防止上下文丢失
